# next-prisma-lead-queue-demo

Демо для портфолио: **публичная форма лида → PostgreSQL → фоновая обработка** отдельным процессом на **TypeScript** (воркер в Docker Compose). Стек: **Next.js 16 (App Router)**, **TypeScript strict**, **Prisma**, **PostgreSQL**. Отдельный REST/NestJS-сервис не используется: граница — **Server Actions** + прямой доступ воркера к БД через Prisma.

## Сквозной сценарий (заказчик)

1. Скопируйте `.env.example` в `.env` и при необходимости поправьте `DATABASE_URL`.
2. Поднимите БД и приложение: `docker compose up --build` (первый запуск применит миграции и соберёт образы).
3. Откройте в браузере `http://localhost:3000`, заполните форму «Оставить заявку» и отправьте.
4. В таблице `Lead` появится запись со статусом `PENDING`; сервис **worker** заберёт её, переведёт в `PROCESSING`, затем в `SUCCEEDED` (поле `processedAt`) или оставит в очереди с ретраями / переведёт в `FAILED` после исчерпания попыток.
5. Повторная отправка с тем же скрытым ключом идемпотентности (UUID в поле `idempotencyKey`) не создаёт дубликат — пользователь видит сообщение о повторной отправке.

**Демо сбоя и ретраев:** укажите email, содержащий подстроку `fail-once@` (например `fail-once@example.com`). Воркер один раз «роняет» внешний шаг и возвращает лид в `PENDING`; со второй попытки обработка завершается успешно (пока не исчерпан `WORKER_MAX_ATTEMPTS`).

## Локально без Docker (опционально)

1. PostgreSQL 16+ с базой `leads` и пользователем из `DATABASE_URL`.
2. `cp .env.example .env` → `npx prisma migrate dev` → `npm install` → `npm run dev` (веб на `http://localhost:3000`).
3. В отдельном терминале: `npm run worker` (те же переменные `DATABASE_URL` и при желании `WORKER_*`).

## Переменные окружения

См. **`.env.example`**. Кратко:

| Переменная | Назначение |
|------------|------------|
| `DATABASE_URL` | Строка подключения Prisma к PostgreSQL |
| `WORKER_POLL_MS` | Пауза между тиками воркера (мс) |
| `WORKER_LOCK_TTL_MS` | Время жизни блокировки; по истечении зависшие `PROCESSING` сбрасываются в `PENDING` |
| `WORKER_MAX_ATTEMPTS` | Максимум попыток обработки одного лида до статуса `FAILED` |

## Архитектура очереди

- **Идемпотентность:** опциональный уникальный `idempotencyKey` (UUID с клиента); гонки закрываются обработкой ошибки уникальности Prisma `P2002`.
- **Статусы:** `PENDING` → `PROCESSING` → `SUCCEEDED` или `FAILED` (с промежуточными возвратами в `PENDING` при ретраях).
- **Ретраи:** счётчик `attemptCount`, лимит `WORKER_MAX_ATTEMPTS`, текст последней ошибки в `lastError`.

## Что докупить под прод

- **Домен и TLS** (хостинг/VPS или PaaS с сертификатами).
- **Управляемый PostgreSQL** (облако) и резервные копии.
- **Наблюдаемость:** логи воркера и веба, алерты по росту `FAILED` и длине очереди `PENDING`.
- **Интеграции:** реальный CRM/webhook вместо `simulateExternalProcess` в `worker/index.ts`, политика PII/GDPR для полей формы.
- **Очередь уровня прод:** при высокой нагрузке — отдельный брокер (Redis, SQS и т.д.); в демо достаточно БД и одного воркера.
